# ROS 2 Bug Algorithm Navigation Tutorial

## Overview

This tutorial demonstrates how to use a map generated by RTAB-Map (from the previous lab) for localization and implement a Bug path planning algorithm (specifically Tangent Bug/Bug 2) for obstacle avoidance.

## Prerequisites

* ROS 2 Humble installed.
* Gazebo simulator installed.
* `rtabmap_ros` packages installed (`sudo apt install ros-humble-rtabmap-ros`).
* `teleop_twist_keyboard` installed (`sudo apt install ros-humble-teleop-twist-keyboard`).
* A working ROS 2 workspace containing the `rtabmap_diff_drive_tutorial` package from the previous tutorial.
* A saved RTAB-Map database file (e.g., `~/.ros/rtabmap.db`) generated during the SLAM phase.

### 2. Path Planning and Bug Algorithms Overview

With the robot localized and sensors providing obstacle information (primarily from the LIDAR now), we can implement path planning. Path planning is the process of finding a sequence of valid configurations (positions and orientations) that moves the robot from a start location to a goal location without colliding with obstacles.

There are many path planning algorithms, broadly categorized into:

* **Global Planners:** These algorithms typically require a complete map of the environment beforehand. They compute an optimal or near-optimal path from start to goal based on the map. Examples include:
  * **Dijkstra's Algorithm:** Finds the shortest path in a graph representation of the map (like an occupancy grid). Guarantees optimality but can be computationally expensive.
  * **A\* (A-Star):** An extension of Dijkstra's that uses a heuristic function to guide the search towards the goal more efficiently. It often finds the shortest path much faster than Dijkstra's while still guaranteeing optimality (if the heuristic is admissible).
  * **Rapidly-exploring Random Trees (RRTs):** Randomized algorithms that are efficient for high-dimensional spaces. They quickly explore the environment by building a tree of possible paths, connecting start to goal. They don't guarantee optimality but are good at finding *a* feasible path quickly.

* **Local Planners (Reactive Planners):** These algorithms react to immediate sensor readings to avoid obstacles. They don't necessarily need a full map beforehand. They are good for dynamic environments or as a component within a larger navigation system (like ROS 2 Nav2) to handle unexpected obstacles not present in the global map.

### 3. Bug Algorithms: Simple Reactive Planning

For this tutorial, we will implement a family of simple, reactive algorithms known as **Bug Algorithms**. These algorithms are intuitive and rely directly on sensor information (like our LIDAR) to navigate around obstacles. They don't require a pre-computed global path from a map, although they do need to know the goal's direction.

The core idea is:

1. Move directly towards the goal.
2. If an obstacle is encountered, follow its boundary (wall following).
3. Apply a specific rule to decide when to leave the obstacle boundary and resume moving towards the goal.

The main variations are:

* **Bug 0:** The simplest. When an obstacle is hit, follow its boundary until the path directly towards the goal is clear again. *Limitation:* Can get trapped in complex concave obstacles.
* **Bug 1:** When an obstacle is hit, circumnavigate it completely, remembering the point on the boundary closest to the goal. Return to that closest point and then continue towards the goal. *Guaranteed to reach the goal if a path exists, but often inefficient.*
* **Bug 2 (Tangent Bug):** When an obstacle is hit, follow its boundary. Continuously check if it's possible to leave the boundary and move directly towards the goal along the line connecting the start and goal points (the "m-line"). Leave the boundary only when on the m-line and closer to the goal than the point where the obstacle was first encountered. *Generally more efficient than Bug 1.*

We will implement a dedicated ROS 2 package. They provide a great introduction to reactive robot control and obstacle avoidance using sensor data.

### 4. Creating the Bug Navigation Package

Create a new ROS 2 package to contain the Bug algorithm implementation.

**a. Create the Package:**

   ```bash
   cd ~/your_workspace/src # Navigate to your workspace src directory
   ros2 pkg create --build-type ament_python bug_navigation --dependencies rclpy geometry_msgs sensor_msgs nav_msgs tf2_ros tf2_geometry_msgs
   ```

**b. Update `package.xml`:**

   ```xml
  <depend>rclpy</depend>
  <depend>geometry_msgs</depend>
  <depend>sensor_msgs</depend>
  <depend>nav_msgs</depend>
  <depend>tf2_ros</depend>       
  <depend>tf2_geometry_msgs</depend> 
   ```

**c. Create Node Script:**

   ```bash
   touch ~/your_workspace/src/bug_navigation/bug_navigation/bug_planner.py
   ```

**d. Update `setup.py`:**

   Add the `entry_points` definition to `bug_navigation/setup.py` to make the node executable.

   ```python
    entry_points={
        'console_scripts': [
         'bug_planner_node = bug_navigation.bug_planner:main',
        ],
    },
   ```

**e. Build the New Package:**

   ```bash
   cd ~/your_workspace
   colcon build --packages-select bug_navigation
   source install/setup.bash
   ```
