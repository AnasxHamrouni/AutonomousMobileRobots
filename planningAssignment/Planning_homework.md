## Task 2: Implement Bug 1 Algorithm

**Goal:** Modify the `bug_navigation` package to implement the Bug 1 algorithm.

**Steps:**

1.  **Create a Copy:** Similar to Task 1, use a separate copy . create a node like `bug1_planner.py`.
2.  **Modify State Machine:** You might need additional state variables to track the circumnavigation process and the closest point found.
3.  **Implement `GO_TO_GOAL`:** Same as before: move towards goal until obstacle hit.
4.  **Implement `WALL_FOLLOW` (Circumnavigation):**
    *   When an obstacle is hit (`obstacle_hit_point`), start following the wall boundary.
    *   **Record the hit point.**
    *   **During wall following:**
        *   Continuously calculate the distance from the *current position* to the *goal*.
        *   Keep track of the position on the boundary (`closest_point_on_boundary`) that had the minimum distance to the goal encountered so far during this circumnavigation. Store both the minimum distance found (`min_dist_to_goal`) and the corresponding point.
    *   **Detect Full Circle:** Determine when the robot has returned to the `obstacle_hit_point` (within a small tolerance) after circumnavigating the obstacle.
5.  **Implement Leave Condition:**
    *   The robot leaves the `WALL_FOLLOW` state **only after completing a full circumnavigation** and returning to the `obstacle_hit_point`.
6.  **Implement New State: `GO_TO_CLOSEST_POINT`:**
    *   After completing the circle, transition to a new state where the robot navigates from the `obstacle_hit_point` towards the stored `closest_point_on_boundary`. You could use the `GO_TO_GOAL` logic, but targeting the `closest_point_on_boundary`.
7.  **Resume `GO_TO_GOAL`:**
    *   Once the robot reaches the `closest_point_on_boundary` (within tolerance), it should then transition back to the original `GO_TO_GOAL` state, now presumably with a clear path.
8.  **Test:** Launch and test with various obstacles. Observe the full circumnavigation behavior. Compare the path length and efficiency to Tangent Bug and Bug 0. Does it always find the goal (if reachable)?

## Task 3: Implement A* (A-Star) Global Planner

**Goal:** Implement a separate ROS 2 node that performs global path planning using the A* algorithm on a map representation.


**Suggested Steps:**

1.  **Map Representation:**
    *   **Obtain Map:** Figure out how to get the occupancy grid map generated by RTAB-Map. Options:
        *   Use RTAB-Map services (like `/rtabmap/get_map`) to retrieve the map data (`nav_msgs/OccupancyGrid`).
        *   Export the map from RTAB-Map database or GUI to a standard format (`.pgm`/`.yaml`) and load it using the `map_server` package.
    *   **Represent Grid:** Load the `nav_msgs/OccupancyGrid` data into a suitable internal representation (e.g., a 2D NumPy array) where you can easily access cell occupancy status (free, occupied, unknown). Handle the conversion from map coordinates to grid cell indices.

2.  **A* Implementation:**
    *   **Data Structures:** Implement necessary structures for A*:
        *   A `Node` or `Cell` class/struct to store cell coordinates (x, y), cost from start (`g`), heuristic cost to goal (`h`), total estimated cost (`f = g + h`), and parent node (for path reconstruction).
        *   An "Open List" to store nodes to be explored, ordered by `f` cost.
        *   A "Closed List" to store nodes already explored.
    *   **Heuristic:** Choose a heuristic function (e.g., Manhattan distance, Euclidean distance) to estimate the cost from a node to the goal.
    *   **Algorithm:** Implement the core A* loop:
        *   Start with the robot's current grid cell in the Open List.
        *   While the Open List is not empty:
            *   Get the node with the lowest `f` cost from the Open List.
            *   If it's the goal node, reconstruct the path by backtracking through parents and return it.
            *   Move the current node to the Closed List.
            *   Explore neighbors (adjacent grid cells):
                *   For each valid neighbor (within grid bounds, not occupied, not in Closed List):
                    *   Calculate tentative `g` cost.
                    *   If the neighbor is not in the Open List or the new path is cheaper:
                        *   Update neighbor's `g`, `h`, `f`, and parent.
                        *   Add/update neighbor in the Open List.

3.  **ROS 2 Integration:**
    *   **Create Node:** Create a new node (e.g., `a_star_planner_node`) in a new package or the `bug_navigation` package.
    *   **Inputs:**
        *   Subscribe to `/map` (if using `map_server`) or use a service client to get the map from RTAB-Map.
        *   Get the robot's current pose (e.g., via TF lookup `map` -> `base_link` or subscribing to `/odom` and using TF).
        *   Subscribe to `/goal_pose`.
    *   **Output:** When a goal is received, run the A* algorithm. Publish the resulting sequence of waypoints as a `nav_msgs/Path` message (e.g., on `/planned_path`).
    *   **Coordinate Conversion:** Carefully handle conversions between world coordinates (meters) and grid cell indices.

4.  **Path Following (Separate Task/Node):**
    *   The A* node only *plans* the path. You would need *another* node (a path follower or local controller) that subscribes to the `/planned_path` and generates `cmd_vel` messages to follow the waypoints, potentially using obstacle avoidance logic similar to `GO_TO_GOAL`. 

5.  **Test:** Launch the map source (map_server or RTAB-Map), your A* planner node, and RViz. Set goals and visualize the `/planned_path` output in RViz. Verify it finds reasonable paths around obstacles in the map.

**Results:** Working A* planner node capable of generating a path (`nav_msgs/Path`) based on a map and start/goal poses. Visualization of the planned path in RViz. Discussion of implementation challenges and heuristic choice.
